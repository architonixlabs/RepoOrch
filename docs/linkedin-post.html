Opening

I've watched this happen too many times.

A support engineer gets a ticket. They have no idea which service changed this morning. So they start pinging developers on Slack.

The developer they ping is on three other things. And they don't fully know either — because the change was made by a different team.

Nobody's at fault. The knowledge just lives in codebases that nobody has time to read right now.

Three People, Same Problem

Support staff are the first to hear when something breaks. But they're often the last to understand why — because that knowledge is buried in a repo they can't navigate and a codebase that changed last Tuesday.

Developers on incident bridges spend the first 30–40 minutes just getting context. Reading code they didn't write. Figuring out what the auth team changed. Meanwhile users are waiting.

New team members tread carefully around services they don't fully understand. Not because they're not capable — but because they genuinely don't know what will break. And nobody has time to explain every contract between every service.

It's not a people problem. It's a context problem.

What I Built

So I built RepoOrch — a Claude Code plugin that does something I haven't seen elsewhere.

Instead of one AI assistant reading one repo, it spawns a team of AI specialists — one per microservice — and makes them actually talk to each other.

You describe a problem. The master agent figures out which services are involved, routes to the right specialists, and each one reads its repo deeply — not just surface files, but contracts, entry points, critical paths, recent churn.

Then they message each other directly. "Does my change to the JWT claim break your service?" The payments specialist responds. They resolve it — or flag it as a risk — before the plan ever reaches you.

At the end, you get one change plan. Ordered. With risks. And nothing is touched without your say-so.

How Deep Each Agent Reads Its Repo
1
Service Registry

Purpose, owners, API version, what it deprecates. The "who is this service" layer.

2
Knowledge Summary

Claude-generated semantic summary: critical paths, entry points, known risks, recent churn. Pre-built so triage is fast.

3
Contract Layer

Auth contracts, error contracts, data contracts, service-level agreements. This is what deliberation is grounded in.

4
Per-Repo Skill File

Human-enriched: gotchas, banned patterns, critical file map, testing conventions. What only your team knows about this service.

5
Live Source Code

The actual files. Read only when the deeper layers don't have the answer. Evidence must cite file + line number.

Safe by Design — Not by Promise

Letting AI agents loose on your codebases is a real concern. So the safety model is enforced at the platform level — not just written in a prompt and hoped for.

🛡️ Prompt injection guard
All file content is treated as data. Instruction-like patterns get stripped before reaching the agents — your repos can't hijack them.
🔒 Write-blocking hook
A platform-level hook hard-blocks all write operations — rm, git commit, git push, file redirects. Enforced, not promised.
📋 Path allowlist
Cross-repo reads are validated against registered repo names. .env files, keys, and credential patterns are blocked by policy.
🧹 Ticket sanitization
Incident descriptions are sanitized before routing. Nobody can slip instructions into a support ticket and redirect the agents.
🗄️ Registry backup & validation
Every registry write is backed up and schema-validated first. Corruption is detected automatically with recovery instructions.
🔍 Stale entry detection
Sync detects registry entries whose repo no longer exists. Prompts to clean them up — no silent drift over time.
Open for Suggestions

I built RepoOrch to scratch my own itch. But the problems it's trying to solve — scattered context, knowledge gaps across teams, the cost of starting from zero on every incident — are bigger than any one tool.

So I'm genuinely curious what I'm missing.

A few things I'm actively thinking about:

Better support for non-Claude toolchains Right now it's tightly coupled to Claude Code. Should it be? What would it look like to work with other AI toolchains?
A read-only mode for support teams Plain-language incident summaries without needing a developer to interpret them — or a CLI to run them.
Slack / ticketing system integration So support staff don't need to run commands at all — triage triggered directly from the ticket.
Confidence scoring per team member Surfacing which parts of the plan are well-evidenced vs. which still need a human eye before anyone acts on them.

If you work with distributed systems — as a developer, support engineer, engineering manager, or architect — I'd genuinely love to hear what would make this more useful for your team.

What's the context problem that costs your team the most time?

Closing

The thing I kept running into while building this: the hard part isn't writing code. It's having enough context to act without second-guessing every decision.

That's true for support staff trying to triage a ticket. It's true for developers joining an incident they didn't cause. And it's especially true for anyone new to a codebase.

RepoOrch doesn't replace your engineers. It just means nobody has to start from zero anymore.

It's open source. Try it, break it, tell me what's missing.
