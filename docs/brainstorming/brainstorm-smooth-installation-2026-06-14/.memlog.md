- (technique) started Five Whys
- (idea) Why1: not working after install because the compiled tools (setup runner, indexer) aren't present — only source ships, they need npm install + tsc
- (idea) Why2: Claude Code plugins ship repo SOURCE, but the value-add tiers are compiled TypeScript needing a build
- (idea) Why3: architecture chose compiled Node packages, dist/ is gitignored, and there is no release/bundle step producing runnable artifacts
- (idea) Why4: the build model was designed for DEVELOPMENT (per-package tsc, gitignored dist) and never reconciled with DISTRIBUTION (installed by non-developers who shouldn't need a toolchain)
- (insight) Five Whys root cause: there is NO packaging boundary between authoring and distribution — no 'ship build' that yields self-contained, dependency-free, install-time artifacts
- (idea) Fix from root: introduce a 'ship build' (esbuild single dependency-free bundle) distinct from the dev build; commit it (setup/dist is NOT gitignored) or attach to release
- (technique) started What If Scenarios
- (idea) What if there were NO build step at all? Ship only prebuilt single-file bundles, or make the installer fully Claude-prompt-native (no compiled code on the critical path)
- (idea) What if Node didn't exist on the machine? The rich runner vanishes — so the Claude-native wizard must be PRIMARY, not the fallback. FLIP the hierarchy
- (idea) What if install were ONE action? A single entry that scans, installs, bootstraps, and restart-gates — collapse setup+init into one guided flow
- (idea) What if the user never edited JSON? Auto-wire MCP into .claude/settings.json, auto-create config.json, programmatic settings merge — zero hand-editing
- (idea) What if the plugin self-verified? A deterministic (no-LLM) post-install health check: files exist, registry schema-valid, agents present -> green/red with auto-fix hints
- (idea) What if install were idempotent by design? Re-running setup is always safe, detects partial state and resumes; add a reset/uninstall command
- (idea) What if restart weren't needed? Detect whether Agent Teams is already active and skip the restart message when it is; otherwise hard restart-gate before triage
- (technique) started Observer Effect
- (insight) Shipping compiled bundles collides with the plugin's transparency/safety ethos — a minified blob is unauditable in a propose-only tool. Ship READABLE (un-minified) bundles + checksum, or prefer the auditable Claude-native path
- (idea) Make the health check deterministic/no-LLM — a real dry-run triage would spend tokens; observing readiness must not cost
- (insight) The smoother the install, the LESS users scrutinize a tool that writes agents+hooks into their workspace — polish erodes propose-only vigilance
- (idea) Turn that around: the wizard SHOWS exactly what it wrote and where as a feature ('wrote 3 agents, 1 read-only hook, never touches service code') + a one-line safety acknowledgement
- (idea) SessionStart hook 'watching' every workspace can false-positive in unrelated monorepos — tighten detection (explicit marker) and make the banner once-per-workspace/dismissible
- (technique) started Irreducible Core (invented, Deep): repeatedly remove install components and ask 'does a useful product still exist?' until you hit the one step that cannot be removed
- (idea) Remove compiled runner -> Claude-native wizard still works (optional). Remove indexer -> Tier-0 works (optional). Remove MCP -> triage works (optional). Remove summaries -> direct reads work (optional). Remove Agent Teams -> single-subagent degraded path
- (insight) Irreducible core of install = 'discover repos -> write valid registry.json + per-repo context'. The registry is the ONE thing install MUST produce; /repo-orch-init alone produces it. So INIT is the irreducible install; SETUP is optional polish
- (insight) BREAKTHROUGH (Irreducible Core + What-If converge): don't FIX the bootstrap paradox — ELIMINATE it by removing the compiled runner from the critical path. Claude-native init is primary & universal; the compiled runner is an optional accelerator that can never block reaching a working registry
- (technique) converged: affinity clustering -> MoSCoW
- (decision) Invert primacy: Claude-native, no-toolchain wizard is the PRIMARY install path (universal + auditable); the bundled compiled runner is an optional speed-up, never on the critical path
- (decision) Packaging boundary: add a 'ship build' (esbuild -> single un-minified dependency-free bundle) for setup runner + indexer; commit setup bundle so it exists at install; keep MCP build-on-install (needs SDK at runtime)
- (decision) Zero hand-editing: installer auto-wires MCP into .claude/settings.json and auto-creates config.json; programmatic JSON merge; user never edits config by hand
- (decision) Transparency-as-feature: wizard prints exactly what it wrote + a one-line safety contract; ship readable (not minified) bundles with a checksum
- (decision) Deterministic post-install health check (no LLM) + idempotent/resumable re-run + restart-gating that is skipped when Agent Teams already active
- (decision) MoSCoW: MUST=invert primacy(Claude-native init core) + auto-wire MCP + fix doc drift + non-blocking tier degradation + restart-gate; SHOULD=@clack/prompts wizard + deterministic health check + 'what I wrote' screen + tighten SessionStart; COULD=two-track runner, dry-run mode, reset cmd; WONT-now=telemetry, minified bundles
- (insight) Two techniques converged on the same point: 'invert primacy' (What If: no Node) and 'init is irreducible' (Irreducible Core) are one realization — the compiled runner must never gate a working install
